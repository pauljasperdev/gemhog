# Phase 4.1: Resend Email Provider - Context

**Gathered:** 2026-01-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace AWS SES email sending with Resend SDK while preserving the existing Effect-based EmailService interface. All existing email functionality (verification, unsubscribe) must work with Resend. SES resources are fully removed.

</domain>

<decisions>
## Implementation Decisions

### Migration strategy
- Full cutover to Resend — remove SES entirely, no fallback
- EmailService interface stays the same — only the Live implementation changes to use Resend SDK
- Remove all SES-related SST infrastructure (no keeping it around unused)
- Local dev continues using mock EmailService — no Resend API key needed locally

### Resend configuration
- Resend account and domain already set up — no setup docs needed in plan
- API key stored as SST Secret (consistent with BETTER_AUTH_SECRET pattern)
- From address: `hello@gemhog.com` for all transactional emails
- Basic sending only — no Resend audiences, broadcasts, or analytics features

### Error handling & deliverability
- Bounce/complaint webhooks deferred to a later phase — basic sending is enough for V1
- Add Resend-specific error types (rate limit, validation, etc.) beyond existing EmailSendError/EmailConfigError
- Retry transient failures (429, 5xx) with exponential backoff before surfacing error
- Send failures reported to Sentry + standard logging

### Testing approach
- Update existing email service tests to verify the new Resend implementation (interface unchanged)
- Use Effect's service layer mocking pattern (consistent with current test approach)
- Automated tests only — no manual verification step in the plan
- No rollback plan — fix forward if issues arise in production

### Claude's Discretion
- Exact Resend SDK version and import patterns
- Retry count and backoff timing
- Resend-specific error type naming and hierarchy
- How to cleanly remove SES infrastructure from SST config

</decisions>
