# Phase 4.1: Resend Email Provider - Research

**Researched:** 2026-01-30
**Domain:** Email sending provider migration (AWS SES to Resend)
**Confidence:** HIGH

## Summary

This phase replaces the AWS SES email sending implementation with the Resend SDK
while preserving the existing Effect-based `EmailService` interface. The codebase
has a clean separation: `EmailServiceTag` defines the interface, and
`makeEmailServiceLive` creates the SES-backed implementation. Only the
implementation factory needs to change.

The Resend Node.js SDK (v6.9.1) provides a simple `resend.emails.send()` API
that returns `{ data, error }` instead of throwing exceptions. It supports custom
headers (including `List-Unsubscribe` already used in the codebase), HTML
content, and TypeScript types out of the box. The SDK accepts an API key via
constructor or `RESEND_API_KEY` env var.

The migration is straightforward because: (1) the `EmailService` interface only
has a `send` method with `to`, `subject`, `html`, and optional `headers` --
exactly the parameters Resend supports; (2) SST infrastructure changes are
limited to removing `infra/email.ts` and swapping the `SES_FROM_EMAIL` env var
for `RESEND_API_KEY`; (3) all existing tests use mocked `EmailService` layers and
do not test the SES implementation directly.

**Primary recommendation:** Replace `@aws-sdk/client-sesv2` with `resend` in
`packages/core`, rewrite `makeEmailServiceLive` to use the Resend SDK, update
env vars from `SES_FROM_EMAIL` to `RESEND_API_KEY`, and remove `infra/email.ts`.

### Pattern 1: Resend-backed EmailService Factory

**What:** Replace the SES-based `makeEmailServiceLive` factory with a
Resend-based one. The factory accepts an API key instead of a sender email
(Resend hardcodes `from` in the send call, not at client creation).

**When to use:** Production environments where `RESEND_API_KEY` is set.

**Example:**
```typescript
// Source: https://resend.com/docs/send-with-nodejs + https://resend.com/docs/api-reference/emails/send-email
import { Resend } from "resend";
import { Console, Context, Effect, Layer, Schedule } from "effect";
import { EmailSendError } from "./email.errors";

export const makeEmailServiceLive = (apiKey: string, fromEmail: string) =>
  Layer.sync(EmailServiceTag, () => {
    const resend = new Resend(apiKey);

    return {
      send: ({ to, subject, html, headers }) =>
        Effect.tryPromise({
          try: async () => {
            const { data, error } = await resend.emails.send({
              from: fromEmail,
              to: [to],
              subject,
              html,
              headers,
            });

            if (error) {
              throw error;
            }

            return data;
          },
          catch: (error) =>
            new EmailSendError({
              message: `Failed to send email to ${to}`,
              cause: error,
            }),
        }),
    };
  });
```

### Pattern 2: Retry with Effect Schedule for Transient Errors

**What:** Wrap the Resend send call with Effect's `retry` using exponential
backoff for transient failures (429 rate limit, 5xx server errors).

**When to use:** Production email sending.

**Example:**
```typescript
// Source: Effect docs + Resend error codes from https://resend.com/docs/api-reference/errors
const retrySchedule = Schedule.exponential("500 millis").pipe(
  Schedule.compose(Schedule.recurs(3)),
);

// Inside the send implementation:
send: ({ to, subject, html, headers }) =>
  sendToResend({ to, subject, html, headers }).pipe(
    Effect.retry({
      schedule: retrySchedule,
      while: (error) => isTransientError(error),
    }),
  ),
```

### Pattern 3: EmailLayers Composition Update

**What:** Update the `EmailLayers` composition to read `RESEND_API_KEY` instead
of `SES_FROM_EMAIL` and use the new factory signature.

**Example:**
```typescript
// apps/web/src/lib/email-layers.ts
import { Layer } from "effect";
import { DatabaseLive } from "@gemhog/core/drizzle";
import {
  EmailServiceConsole,
  makeEmailServiceLive,
  SubscriberServiceLive,
} from "@gemhog/core/email";
import { env } from "@gemhog/env/server";

export const EmailLayers = Layer.mergeAll(
  env.RESEND_API_KEY
    ? makeEmailServiceLive(env.RESEND_API_KEY, "hello@gemhog.com")
    : EmailServiceConsole,
  SubscriberServiceLive.pipe(Layer.provide(DatabaseLive)),
);
```

### Anti-Patterns to Avoid

- **Instantiating Resend client per send call:** Create the `Resend` instance
  once in the Layer factory (`Layer.sync`), not inside the `send` method. The
  current SES pattern already does this correctly.

- **Throwing on API errors:** Resend SDK returns `{ data, error }` and does NOT
  throw for API errors (validation, rate limits, etc.). You must check the
  `error` field. Only network-level failures throw.

- **Using `RESEND_API_KEY` env var implicitly:** The Resend SDK auto-reads
  `RESEND_API_KEY` from the environment if no key is passed to the constructor.
  Do NOT rely on this -- pass the key explicitly via the factory for consistency
  with the project's env validation pattern (`@gemhog/env`).

## Don't Hand-Roll

| Problem                  | Don't Build                 | Use Instead             | Why                                                  |
| ------------------------ | --------------------------- | ----------------------- | ---------------------------------------------------- |
| Email sending            | Raw HTTP to Resend API      | `resend` npm package    | TypeScript types, error parsing, auth header handling |
| Retry logic              | Custom retry loops          | `Effect.retry` + `Schedule` | Already using Effect; composable and testable     |
| Rate limit handling      | Manual delay/queue          | `Effect.retry` with `Schedule.exponential` | Built into Effect; handles backoff correctly |

**Key insight:** The Resend SDK handles HTTP communication, auth headers, and
response parsing. Effect handles retry, error types, and DI. There is no gap
requiring custom code beyond wiring them together.

## Common Pitfalls

### Pitfall 1: Resend SDK Error Return Pattern

**What goes wrong:** Treating Resend SDK calls like traditional throw-on-error
libraries. The SDK returns `{ data, error }` where `error` is populated on API
failures instead of throwing.

**Why it happens:** Most HTTP client libraries throw on non-2xx responses. Resend
SDK deliberately returns errors as values.

**How to avoid:** Always destructure `{ data, error }` from the send call and
check `error` before using `data`. In the Effect wrapper, throw the error inside
`tryPromise.try` to convert it to `EmailSendError`.

**Warning signs:** Emails silently "succeed" but never arrive. No error logging.

### Pitfall 2: DNS Records Conflict

**What goes wrong:** After removing SES infrastructure, the SES-created DNS
records (DKIM CNAME, SPF TXT, DMARC TXT) remain in Cloudflare. Resend requires
its own DNS records. Conflicting records cause deliverability issues.

**Why it happens:** SST's `sst.aws.Email` component creates DNS records via
Cloudflare DNS adapter. Removing the component should clean up its records, but
Resend's domain verification adds new records through the Resend dashboard.

**How to avoid:** Verify that the SES DNS records are removed after
`sst deploy`. Then add Resend's DNS records via the Resend dashboard. The domain
`gemhog.com` is already set up in Resend per the context decisions.

**Warning signs:** Domain verification fails in Resend dashboard. SPF includes
both `amazonses.com` and Resend origins.

### Pitfall 3: `from` Address Format

**What goes wrong:** Resend supports `"Name <email@domain.com>"` format for the
`from` field. Using just `email@domain.com` works but shows no display name.

**Why it happens:** The current SES implementation uses `senderEmail` as a plain
email address string (`hello@gemhog.com`).

**How to avoid:** Use `"Gemhog <hello@gemhog.com>"` as the `from` value to show
a friendly name in recipients' inboxes.

**Warning signs:** Emails show raw email address as sender name.

### Pitfall 4: Factory Signature Change Breaks Callers

**What goes wrong:** The current `makeEmailServiceLive(senderEmail)` takes one
argument. The new version needs both API key and from address. Callers that pass
`env.SES_FROM_EMAIL` will fail at compile time.

**Why it happens:** Different email providers have different configuration
requirements.

**How to avoid:** Update ALL call sites: `email-layers.ts` in `apps/web` and
`getEmailLayers()` in `packages/api/src/routers/subscriber.ts`. There are
exactly two call sites in the codebase.

**Warning signs:** TypeScript compilation errors (caught at build time).

### Pitfall 5: Rate Limit (2 req/sec Default)

**What goes wrong:** Resend's default rate limit is 2 requests per second. Burst
sending (e.g., future newsletter sends) hits 429 errors.

**Why it happens:** Resend applies conservative default rate limits to new
accounts.

**How to avoid:** For Phase 4.1 (transactional emails only -- verification,
unsubscribe), 2 req/sec is sufficient. Add retry with exponential backoff to
handle occasional 429s. Newsletter sends are deferred to V2.

**Warning signs:** `rate_limit_exceeded` errors in Sentry.

## Code Examples

### Complete Resend EmailService Implementation

```typescript
// Source: https://resend.com/docs/send-with-nodejs
//         https://resend.com/docs/api-reference/errors
import { Resend } from "resend";
import { Context, Effect, Layer, Schedule } from "effect";
import { EmailSendError, ResendRateLimitError } from "./email.errors";

export interface SendEmailParams {
  to: string;
  subject: string;
  html: string;
  headers?: Record<string, string>;
}

export interface EmailService {
  readonly send: (
    params: SendEmailParams,
  ) => Effect.Effect<void, EmailSendError>;
}

export class EmailServiceTag extends Context.Tag("EmailService")<
  EmailServiceTag,
  EmailService
>() {}

// Console implementation (local dev) -- unchanged
export const EmailServiceConsole = Layer.succeed(EmailServiceTag, {
  send: (params) =>
    Effect.gen(function* () {
      // ... existing console logging code unchanged ...
    }),
});

// Resend implementation (production)
export const makeEmailServiceLive = (apiKey: string, fromEmail: string) =>
  Layer.sync(EmailServiceTag, () => {
    const resend = new Resend(apiKey);

    return {
      send: ({ to, subject, html, headers }) => {
        const sendEffect = Effect.tryPromise({
          try: async () => {
            const { data, error } = await resend.emails.send({
              from: fromEmail,
              to: [to],
              subject,
              html,
              headers,
            });

            if (error) {
              throw error;
            }

            return data;
          },
          catch: (error) =>
            new EmailSendError({
              message: `Failed to send email to ${to}`,
              cause: error,
            }),
        });

        // Retry transient failures (429, 5xx) with exponential backoff
        const retrySchedule = Schedule.exponential("500 millis").pipe(
          Schedule.compose(Schedule.recurs(3)),
        );

        return sendEffect.pipe(
          Effect.retry({
            schedule: retrySchedule,
            while: (error) => isTransientError(error),
          }),
          Effect.asVoid,
        );
      },
    };
  });

function isTransientError(error: EmailSendError): boolean {
  const cause = error.cause as { name?: string; statusCode?: number } | undefined;
  if (!cause) return false;
  // Resend error types: rate_limit_exceeded (429), application_error (500), internal_server_error (500)
  const transientNames = ["rate_limit_exceeded", "daily_quota_exceeded", "monthly_quota_exceeded", "application_error", "internal_server_error"];
  return transientNames.includes(cause.name ?? "") || (cause.statusCode ?? 0) >= 500;
}
```

## Open Questions

1. **`CloudflareZoneId` secret after removing SES Email component**
   - What we know: `CloudflareZoneId` is used in `infra/email.ts` (for SES DNS)
     and `infra/router.ts` (for API/web DNS). It's also used in `infra/web.ts`
     (for web domain DNS).
   - What's unclear: Nothing -- the secret is still needed for
     `infra/router.ts` and `infra/web.ts`.
   - Recommendation: Keep `CloudflareZoneId` in secrets. Only remove the email
     infrastructure.

2. **`from` address hardcoding**
   - What we know: Context decision says `from` is `hello@gemhog.com`. This
     could be hardcoded in the factory call or passed as an env var.
   - What's unclear: Whether to make it configurable via env or hardcode.
   - Recommendation: Hardcode `"Gemhog <hello@gemhog.com>"` in the layer
     composition (email-layers.ts / subscriber.ts). A single from address for
     all transactional emails is standard. If it needs to change, it's a code
     change, not a config change -- acceptable for V1.
